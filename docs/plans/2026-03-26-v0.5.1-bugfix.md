### Step 1: Fix TID mismatch unbounded loop (S1)

Add a stale-discard counter before the `loop {` at line 1025. When more than `MAX_STALE_RESPONSES = 5` frames are discarded, return an error instead of looping forever.

Find the loop:
```rust
let response_buf = loop {
    // Read response header first (MBAP header + function code) into persistent buf
    let read_result = timeout(
        self.timeout,
        stream.read_exact(&mut self.read_buf[..MBAP_HEADER_SIZE + 1]),
    )
    .await;
```

Replace with:
```rust
const MAX_STALE_RESPONSES: usize = 5;
let mut stale_count = 0usize;
let response_buf = loop {
    if stale_count >= MAX_STALE_RESPONSES {
        self.stats.errors += 1;
        self.stream = None;
        return Err(ModbusError::protocol(
            "too many mismatched responses; possible bus conflict",
        ));
    }
    let read_result = timeout(
        self.timeout,
        stream.read_exact(&mut self.read_buf[..MBAP_HEADER_SIZE + 1]),
    )
    .await;
```

Then increment `stale_count` on each `continue`:
```rust
// Before the two `continue` statements at lines 1111 and 1122:
stale_count += 1;
continue;
```

### Step 1: Fix PDU byte_count overflow (S3)

In `src/pdu.rs` at line ~351, `build_write_multiple_registers`:
```rust
// OLD:
let byte_count = (values.len() * 2) as u8;

// NEW:
let byte_count = u8::try_from(values.len().checked_mul(2).unwrap_or(usize::MAX))
    .map_err(|_| ModbusError::invalid_data("too many register values for PDU (max 123)"))?;
```

Also add an explicit bounds check before the byte_count line:
```rust
if values.len() > 123 {
    return Err(ModbusError::invalid_data(
        "write_multiple_registers: maximum 123 registers per request",
    ));
}
let byte_count = (values.len() * 2) as u8; // safe: 123 * 2 = 246 < 256
```

Similarly fix `build_write_multiple_coils` if it has the same pattern (search for `as u8` near byte_count).

### Step 2: Fix `data.len() as u8` in transport encode (S7)

Search in `src/transport.rs` for all occurrences of `request.data.len() as u8` (lines ~647, ~761, ~1362, ~1371, ~1892). For each one, wrap in a checked cast:

```rust
// OLD:
frame[pos] = request.data.len() as u8;

// NEW:
frame[pos] = u8::try_from(request.data.len()).map_err(|_| {
    ModbusError::invalid_data("data payload too large for Modbus frame")

```

Note: These are in `encode_request` / `encode_request_with_tid` for `WriteMultipleCoils` and `WriteMultipleRegisters`. Because high-level client API already validates lengths, this protects the raw `ModbusRequest` API path.

```rust
#[tokio::test]
async fn test_read_coalesced_end_to_end() {
    // Set up MockTransport that handles a single merged FC03 read
    // Requests: (addr=0, qty=2), (addr=2, qty=2), (addr=10, qty=2)
    // Coalescer merges first two: (addr=0, qty=4), third is separate: (addr=10, qty=2)
    // We verify:
    //   1. Exactly 2 transport requests are made (one merged, one separate)
    //   2. Returned values are extracted from the correct offsets

    let mut mock = MockTransport::new();
    // Response for merged request (addr=0, qty=4): registers [1, 2, 3, 4]
    mock.add_response(1, 0x03, 0, 4, vec![0,1, 0,2, 0,3, 0,4]);
    // Response for separate request (addr=10, qty=2): registers [10, 11]
    mock.add_response(1, 0x03, 10, 2, vec![0,10, 0,11]);

    let mut client = GenericModbusClient::new(mock);

    let requests = vec![
        ReadRequest::new(0, 2),
        ReadRequest::new(2, 2),
        ReadRequest::new(10, 2),
    ];
    let results = client.read_holding_registers_coalesced(1, &requests).await.unwrap();

    assert_eq!(results.len(), 3);
    assert_eq!(results[0], vec![1u16, 2]);
    assert_eq!(results[1], vec![3u16, 4]);
    assert_eq!(results[2], vec![10u16, 11]);

    // Verify only 2 transport requests were issued (coalescing worked)
    assert_eq!(client.transport().request_count(), 2);
}
```

Note: This test requires `MockTransport` to track request count and support address-based response matching. Adjust to match the actual `MockTransport` API in the test module.

### Step 4: test_pipeline_out_of_order_responses (src/client.rs)

The existing pipeline tests use `spawn_mock_server`. Add a test that sends responses in reverse TID order:

```rust
#[tokio::test]
async fn test_pipeline_out_of_order_responses() {
    use tokio::net::TcpListener;
    use tokio::io::{AsyncReadExt, AsyncWriteExt};

    let listener = TcpListener::bind("127.0.0.1:0").await.unwrap();
    let addr = listener.local_addr().unwrap();

    // Server: accept one connection, read all requests, reply in REVERSE order
    tokio::spawn(async move {
        let (mut stream, _) = listener.accept().await.unwrap();
        // Read two FC03 requests (12 bytes each)
        let mut req1 = [0u8; 12];
        let mut req2 = [0u8; 12];
        stream.read_exact(&mut req1).await.unwrap();
        stream.read_exact(&mut req2).await.unwrap();

        // Extract TIDs
        let tid1 = u16::from_be_bytes([req1[0], req1[1]]);
        let tid2 = u16::from_be_bytes([req2[0], req2[1]]);

        // Reply to request 2 first (reverse order), then request 1
        let resp2 = build_fc03_response(tid2, 1, &[200u16, 201]);
        let resp1 = build_fc03_response(tid1, 1, &[100u16, 101]);
        stream.write_all(&resp2).await.unwrap();
        stream.write_all(&resp1).await.unwrap();
    });

    let mut client = ModbusTcpClient::from_address(
        &addr.to_string(), Duration::from_secs(2)
    ).await.unwrap();

    let requests = vec![
        ModbusRequest { slave_id: 1, function: ModbusFunction::ReadHoldingRegisters,
                        address: 0, quantity: 2, data: vec![] },
        ModbusRequest { slave_id: 1, function: ModbusFunction::ReadHoldingRegisters,
                        address: 10, quantity: 2, data: vec![] },
    ];

    let results = client.pipeline(requests, Duration::from_secs(2)).await.unwrap();
    assert_eq!(results.len(), 2);
    // Results must be in original request order regardless of response order
    let regs0 = results[0].data_as_registers().unwrap();
    let regs1 = results[1].data_as_registers().unwrap();
    assert_eq!(regs0, vec![100u16, 101], "first result should be for first request");
    assert_eq!(regs1, vec![200u16, 201], "second result should be for second request");
}
```

Note: Implement `build_fc03_response` helper in the test module to construct a valid MBAP-framed FC03 response.
